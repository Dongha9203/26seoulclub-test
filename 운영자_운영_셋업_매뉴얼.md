## 2. Vercel에 배포하기 (최초 1회)

1. [vercel.com](https://vercel.com) 에 본인(운영자) 계정으로 로그인
2. **Add New...** → **Project**
3. GitHub 저장소 `seoulclub-on/26seoulclub` 선택해서 Import
   - 목록에 안 보이면: GitHub → **Settings** → **Applications** → **Vercel** → **Configure**에서
     Repository access에 `26seoulclub`이 추가되어 있는지 확인 (없으면 추가 후 Save)
4. Framework Preset: 자동 감지 안 되면 **Other** 선택
5. **Environment Variables**에 1번에서 확인한 환경변수 값을 모두 입력
   (Production / Preview / Development 전부 체크)
6. **Deploy** 클릭
7. 배포 완료 후 나온 주소(예: `https://26seoulclub.vercel.app`) 확인

## 4. 운영 중 알고 있어야 할 것

### a) 노션에 새 정보를 추가할 때
- 챗봇은 **"메인페이지(2026 서울 동아리ON)"** 1개를 최상위로 수집하며, 그 안에 있는
  모든 하위 페이지(예: "통합시스템(필독)")도 자동으로 함께 수집됩니다.
  하위 페이지/토글/섹션을 추가하는 건 자동으로 챗봇에 반영됩니다(매일 새벽 자동
  갱신 또는 대시보드 "지금 갱신" 버튼).

> **참고 — config.json의 `integrated_system` 항목이 비어 있는 이유**
>
> "통합시스템(필독)" 노션 페이지는 메인페이지의 하위 페이지로 이미 포함되어 있습니다.
> 초기에는 이 페이지 URL을 `config.json`의 `integrated_system`에 별도로 등록했으나,
> 메인페이지 수집 시 한 번, `integrated_system` URL로 한 번, 총 2번 수집되어
> DB에 중복 저장되는 문제가 확인되었습니다.
> 이를 방지하기 위해 `integrated_system` 값을 `""`(빈 값)으로 변경하였으며,
> 이후에는 메인페이지 탐색 과정에서 1번만 수집됩니다.
> **이 빈 값을 다시 채우지 마세요.**
- 단, 그 페이지가 **다른 사람이 새로 만들었거나 워크스페이스 다른 곳에서 옮겨온
  페이지**라면, 그 페이지를 열어 우측 상단 **"..." → "연결(Connections)"**에
  챗봇 통합이 추가되어 있는지 꼭 확인하세요. 빠지면 챗봇이 그 내용을 영원히
  찾지 못합니다. (실제로 "강사비 지급 기준" 페이지가 이 문제였습니다.)
- 완전히 새로운 **최상위** 페이지(메인페이지/통합시스템과 같은 레벨)를 추가로
  연결하려면 `config.json`의 `notion_pages` 항목에 새 URL을 추가하고 Vercel에
  재배포하면 됩니다. (4-e 항목 참고)
- **Airtable과 노션 데이터베이스는 서로 다른 기능**이고, 챗봇이 읽는 범위도 다릅니다.
  - 노션 페이지에 **Airtable 표/폼을 임베드**해 넣은 경우 → 표 안 데이터는
    못 읽고, 그 자리에 있던 링크만 "관련 링크: https://..." 형태로 검색에 남습니다.
  - 노션 자체의 **데이터베이스(표 보기) 기능**으로 만든 자료(예: "자료실")는
    각 행(row)을 별도 페이지로 보고 그 안의 내용을 챗봇이 읽어 검색에
    활용합니다. 행 추가/삭제/수정도 매일 새벽 자동 갱신에 반영됩니다.
- **이미지·첨부파일은 챗봇이 읽을 수 없습니다.**
  - 노션 페이지에 이미지(사진, 스크린샷, 안내 이미지 등)나 PDF·파일 첨부로만
    올라와 있는 정보는 챗봇이 내용을 인식하지 못합니다.
  - 이미지 안에 텍스트(표, 안내문 등)가 담겨 있어도 마찬가지로 읽지 못하며,
    해당 내용에 대한 질문에는 답변이 불가합니다.
  - **해결 방법**: 이미지로만 제공된 중요 정보는 노션 페이지에 **텍스트로 함께
    작성**해두면 챗봇이 검색해 답변할 수 있습니다.

### b-1) Vercel Pro 플랜 업그레이드 시 추가되는 기능 (현재는 무료 플랜 운영 중)

현재는 Vercel 무료(Hobby) 플랜으로 운영합니다. 아래는 Pro 플랜($20/월)으로 업그레이드할 경우 추가되는 기능입니다. 필수는 아니지만, 운영 규모가 커지면 고려할 수 있습니다.

| 기능 | 무료(Hobby) | Pro |
|------|------------|-----|
| **콜드 스타트 방지** | 없음 — 미사용 시 항상 발생 | **"Scale to One"**: 마지막 사용 후 14일간 서버 1개 유지 |
| 자동 크론 실행 빈도 | 하루 1번 | 분 단위 가능 (예: 매 10분, 매 시간) |
| Vercel 런타임 로그 보존 기간 | **1시간** | **3일** |
| Vercel 대시보드 접근 계정 수 | 본인 1명 | 팀 멤버 초대 가능 |

**각 기능이 필요해지는 시점:**
- **콜드 스타트 방지**: 챗봇 첫 응답이 10초 이상 걸리는 문제를 개선하고 싶을 때. Pro로 업그레이드하면 마지막 사용 후 14일 이내에는 서버가 켜진 채 유지되어 콜드 스타트가 발생하지 않습니다. 단, 14일 이상 아무도 사용하지 않으면(예: 긴 방학) 다시 콜드 스타트가 발생합니다. (콜드 스타트가 무엇인지는 아래 4-j 항목 참고)
  챗봇과 관리자 대시보드는 **같은 서버 인스턴스**에서 동작하기 때문에, Cold Start 방지 효과는 챗봇 응답 속도뿐 아니라 **대시보드 첫 접속 속도**에도 동일하게 적용됩니다. 오랜 미사용 후 운영자가 대시보드를 처음 열 때도 Cold Start가 없어져 메뉴가 바로 로드됩니다.
- **크론 빈도**: 노션에 자료를 올리고 챗봇에 빠르게 반영되어야 할 때 (현재는 하루 1번 또는 수동 "지금 갱신"으로 운영 중)
- **로그 보존 3일**: 아래 두 단계로 크론 이상을 확인할 수 있습니다.
  - **1단계 — 어드민 패널 배너**: 로그인 후 사이드바의 크론 상태 배너(초록/노랑)로 마지막 실행 시각·성공 여부를 빠르게 확인
  - **2단계 — Vercel 런타임 로그**: 배너에서 이상이 감지되거나(노랑/빨강), 배너는 정상인데 챗봇 답변이 이상할 때, 원인을 코드 레벨에서 추적하려면 Vercel 대시보드의 런타임 로그가 필요합니다. **무료 플랜은 로그가 1시간만 보존**되므로 새벽 4시 크론 로그를 오전에 확인하려 해도 이미 사라진 상태입니다. Pro 플랜(3일 보존)이라면 당일 중 언제든 로그를 열어볼 수 있습니다. (크론 이상 시 전체 대처 절차는 유지보수 매뉴얼 6-b 참고)
- **팀 멤버**: 운영팀 여러 명이 Vercel 대시보드에 직접 접근해야 할 때

### g) 지식DB(Knowledge Base) 노후화 관리

챗봇이 답변의 근거로 쓰는 문서(지식DB)는 시간이 지나면서 현실과 어긋날 수 있습니다.

- **노후화가 발생하는 이유**: 행사가 종료되거나 일정이 바뀌어도 노션 페이지에 내용이 그대로 남아 있으면 챗봇이 오래된 정보를 계속 안내합니다. 담당자 연락처 변경, 제출 폼 URL 변경, 활동 기준 변경 등도 마찬가지입니다.

- **운영자가 해야 할 일**:
  - 프로그램 일정이 끝났거나 내용이 바뀌었을 때, 해당 노션 페이지를 직접 수정하거나 삭제한 뒤 대시보드 "지금 갱신"을 눌러 챗봇에 반영하세요.
  - 파일·구글시트로 등록한 문서도 주기적으로 점검해 만료된 문서는 삭제하고 최신 문서로 교체하세요.
  - 대시보드 "원인별 집계 리포트"에서 "검색실패" 건수가 갑자기 늘어나면, 지식DB에 최신 정보가 반영되어 있는지 점검하세요.

- **자동 갱신의 한계**: 매일 새벽 자동 동기화는 "노션에 있는 내용을 챗봇에 반영"하는 것이지, 오래된 내용을 스스로 판단해 삭제하지 않습니다. 노션 페이지를 운영자가 최신 상태로 직접 관리하는 것이 전제입니다.

### h) Supabase DB 용량 제한

챗봇의 모든 데이터(지식 문서, 임베딩 벡터, 질의응답 기록 등)는 Supabase PostgreSQL에 저장됩니다. Supabase 무료(Free) 플랜의 DB 저장 한도는 **500 MB**입니다.

**한도 초과 시**: Supabase가 해당 프로젝트를 자동 일시정지합니다. 일시정지되면 챗봇과 대시보드가 모두 중단되며, Supabase 대시보드에서 수동으로 복구해야 합니다. (복구 절차는 유지보수 매뉴얼 6-c 참고)

**자동 알림**: Supabase는 사용량이 75%(375 MB), 90%(450 MB)에 도달하면 프로젝트 소유자 이메일로 자동 경고 메일을 발송합니다. 별도 설정 없이 활성화되어 있으나, 프로젝트를 만든 계정의 이메일 수신함을 실제로 확인하는 것이 전제입니다.

**수동 확인 방법**: [supabase.com](https://supabase.com) → 해당 프로젝트 → **Settings → Usage** 에서 현재 사용량을 직접 확인할 수 있습니다.

**용량을 주로 차지하는 항목**:

| 테이블 | 내용 |
|--------|------|
| `documents` | 지식DB 문서 + 임베딩 벡터 (인덱스가 데이터 대비 크게 차지) |
| `qa_log` | 질의응답 기록 (90일 이후 자동 삭제) |
| `cron_run_log` | 자동 동기화 실행 기록 |

지식DB 문서는 수동으로 관리하지 않으면 계속 누적됩니다. 더 이상 사용하지 않는 파일·구글시트 문서는 주기적으로 정리하세요. (SQL Editor를 통한 직접 정리는 유지보수 매뉴얼 6-d 참고)

### i) 챗봇 개인정보 처리

**수집되는 정보**: 사용자가 챗봇 위젯에 입력한 질문 텍스트, 챗봇의 답변, 실패 원인 분류, 응답 속도, 세션 ID가 `qa_log` 테이블에 저장됩니다. 이름·연락처 등 신원 정보는 별도로 요청하거나 수집하지 않지만, 사용자가 질문 내용에 직접 포함시킬 경우 그 내용도 로그에 남습니다.

**세션 식별**: 동일 사용자 판별에 브라우저 세션 ID를 사용합니다. 로그인 없이 익명으로 동작하며, 세션 ID는 브라우저의 localStorage에 저장되므로 탭/창을 닫아도 유지됩니다. 같은 브라우저에서 다시 접속하면 동일 세션으로 이어집니다.

**보관 기간**: 질의응답 기록은 **90일 후 자동 삭제**됩니다.

**열람 가능 범위**: 대시보드 계정을 가진 운영자는 "질의-답변 연계조회" 메뉴에서 최근 90일 이내의 모든 질의응답 내용을 열람할 수 있습니다.

**운영자가 확인해야 할 사항**:
- 챗봇 위젯이 임베드된 노션 페이지에서, 질문 내용이 로그에 기록된다는 사실을 사용자에게 안내하는 것이 적절한지 판단하세요.
- 대시보드 접근 계정(이메일·비밀번호)이 외부에 유출되지 않도록 관리하세요. 대시보드에서 모든 질의응답 내용을 열람할 수 있습니다.
- 의심스러운 로그인 시도가 확인되면 즉시 비밀번호를 변경하세요.

### j) 챗봇 응답 속도 이해 — Cold Start와 Warm Start

챗봇에 질문했을 때 가끔 첫 응답이 **10초 이상** 걸리는 경우가 있습니다. 이는 시스템 오류가 아니라 **Cold Start(콜드 스타트)** 현상입니다.

#### 식당에 비유한 설명

| 상황 | 비유 | 챗봇 |
|------|------|------|
| **Warm Start (정상 속도)** | 주방이 열려 있고 요리사가 대기 중 → 주문 즉시 요리 시작 | 서버가 켜져 있음 → 질문 즉시 처리 (약 2~4초) |
| **Cold Start (느린 첫 응답)** | 식당 문이 잠겨 있음 → 문 열고, 주방 기기 켜고, 재료 꺼내고, 요리사 자리 잡은 후 → 그제야 주문 접수 | 서버가 꺼져 있음 → 다시 켜는 시간 필요 (약 8~15초) |

#### 왜 이 챗봇의 Cold Start가 특히 오래 걸리나

서버가 켜질 때 아래 작업들이 자동으로 순서대로 실행됩니다:

```
서버 시작
  ├─ 한국어 형태소 분석기(kiwipiepy) 등 무거운 프로그램 로딩  ── 약 3~5초
  ├─ 지식DB(질문에 답할 문서들) 메모리에 올리기             ── 약 5초
  ├─ AI 검색 서버(Voyage AI) 연결 준비                    ── 약 1.5초
  └─ 완료 후 → 그제야 첫 질문 처리 가능
```

**결과:** Cold Start 총 소요 시간 **약 8~15초** / Warm Start 응답 시간 **약 2~4초**

#### Cold Start가 발생하는 시점 (현재 무료 플랜 기준)

- 서버가 일정 시간 동안 **아무 요청도 없으면 자동으로 꺼집니다.**
- 그 다음 서버로 들어오는 첫 번째 요청(챗봇 질문이든 대시보드 접속이든)에서 서버를 다시 켜면서 Cold Start가 발생합니다.
- 예를 들어 야간에 아무도 사용하지 않았다면, 다음날 **첫 번째로 질문하는 학생**이 Cold Start를 경험하고, 그보다 먼저 운영자가 대시보드를 열었다면 **운영자의 첫 접속**에서 Cold Start가 발생합니다.

#### 개선 방법 요약

| 방법 | 효과 | 비용 |
|------|------|------|
| **현재 (무료 Vercel)** | Cold Start 발생 가능 | 무료 |
| **Vercel Pro 업그레이드** | 마지막 사용 후 14일간 서버 유지 → 학기 중 Cold Start 거의 없음. 단 방학 2주 이상 시 여전히 발생 | 월 $20 |
| **Railway로 서버 변경** | 방학·비수기 구분 없이 **서버가 항상 켜져 있음** → Cold Start 원천 차단 | 월 약 $5~10 |

두 가지 개선 방법의 상세 안내는 **섹션 6** 을 참고하세요.

### f) Voyage AI 결제수단 등록 (필수 — 미등록 시 챗봇 정상 운영 불가)

> **반드시 배포 전에 완료해야 합니다. 결제수단을 등록하지 않으면 챗봇이 정상적으로 동작하지 않습니다.**

결제수단이 등록되지 않은 Voyage AI 계정은 **분당 최대 3건** 요청으로 제한됩니다.
이 제한은 아래 두 가지 상황 모두에 영향을 줍니다.

**① 사용자 채팅 응답 실패**
챗봇이 질문에 답변하려면 매 질문마다 Voyage AI API를 1회 호출합니다.
분당 3건 제한 상태에서 동시에 여러 사용자가 질문하면 초과 요청은 즉시 실패하고,
해당 사용자는 답변 대신 "운영팀에 문의하세요" 안내만 받습니다.
몇 명이 동시에 접속하느냐에 따라 언제든 발생할 수 있으며, 사용자 수가 늘수록 빈도도 높아집니다.

**② 노션 동기화 시 임베딩 누락**
동기화 시 한 번에 갱신되는 문서가 많으면 제한에 걸려 일부 문서가
검색에 반영되지 않고 조용히 실패합니다(에러 화면 없이 그냥 안 보임).

**등록 방법**: [Voyage AI 대시보드](https://dashboard.voyageai.com) → Billing 페이지에서 결제수단 등록.
결제수단을 등록해도 무료 토큰은 그대로 유지되며,
무료 토큰 소진 전까지는 실제 청구 금액이 발생하지 않습니다.
(현재 사용 모델: `voyage-4`. 정확한 무료 토큰 잔량은 [Voyage AI 대시보드](https://dashboard.voyageai.com) → Usage 탭에서 확인하세요.)

> Cold Start가 무엇인지 모른다면 **4-j 항목**을 먼저 읽으세요.

현재 Vercel 무료(Hobby) 플랜에서는 서버가 일정 시간 미사용 시 자동으로 꺼지기 때문에

## 옵션 A: Vercel Pro 업그레이드

> 현재 Vercel 환경을 그대로 유지하면서 플랜만 변경합니다.
> 추가 설정이나 마이그레이션 없이 **5분** 안에 완료됩니다.

### A-1) 효과와 한계

**좋아지는 것**
- 마지막 사용 후 **14일 이내**에는 서버가 켜진 채 유지 → Cold Start 발생 안 함
- 학기 중처럼 학생들이 꾸준히 사용하는 기간에는 Cold Start가 사실상 없어집니다

**여전히 남는 한계**
- 방학처럼 **14일 이상 아무도 사용하지 않으면** 서버가 꺼지고 Cold Start 재발
- 새 배포(코드 수정 후 재배포) 직후 첫 요청은 항상 Cold Start 발생

### A-2) 업그레이드 방법 (약 5분)

1. [vercel.com](https://vercel.com) 접속 → 우측 상단 계정 아이콘 클릭
2. **"Plans"** 또는 **"Upgrade"** 선택
3. **Pro 플랜** 선택 → 결제 정보 입력 → 완료
4. 별도 재배포 없이 즉시 적용됩니다

> Pro 플랜은 **월 $20/인**입니다. Cron 빈도, 로그 보존 기간 등
> Cold Start 외 추가 혜택은 **4-b-1** 항목을 참고하세요.

## 옵션 B: Railway로 서버 변경

> Vercel 대신 Railway 서버로 이전하면 방학·비수기 구분 없이
> **서버가 항상 켜져 있어 Cold Start가 발생하지 않습니다.**
> 단, 아래 마이그레이션 작업이 필요합니다. (약 55분)

### B-1) Railway와 Vercel의 근본적인 차이

```
[Vercel 방식]
  사용자 없음 → 서버 자동 꺼짐 → 사용자 접속 → 서버 다시 켬 (Cold Start)

[Railway 방식]
  사용자 없음 → 서버 그대로 켜져 있음 → 사용자 접속 → 즉시 응답
```

> **참고**: Railway는 소규모 내부 도구 수준에서 안정적으로 운영됩니다.
> 단, 간헐적인 배포 실패나 지원 응답 지연 사례가 보고된 적 있으니 참고하세요.

---

### B-2) 마이그레이션 전 준비사항

현재 Vercel에 입력된 환경변수 5개를 미리 메모장에 복사해둡니다.

**복사 방법**: Vercel 대시보드 → 해당 프로젝트 → **Settings** → **Environment Variables**

| 복사할 변수명 | 용도 |
|--------------|------|
| `SUPABASE_DB_URL` | DB 연결 주소 |
| `NOTION_API_TOKEN` | 노션 자동 동기화 인증 키 |
| `ANTHROPIC_API_KEY` | Claude AI 인증 키 |
| `VOYAGE_API_KEY` | 검색 AI 인증 키 |
| `JWT_SECRET_KEY` | 대시보드 로그인 인증 키 |

> **주의**: 이 값들은 외부에 절대 노출하지 마세요.
> 카카오톡·이메일 등에 평문으로 저장하지 말고 비밀번호 관리 앱을 사용하세요.

### B-3) Railway 가입 및 프로젝트 생성 (약 10분)

### B-5) 서버 시작 명령어 설정 (약 5분)

Railway는 서버를 어떻게 실행할지 알려줘야 합니다.

1. Railway 대시보드 → 프로젝트 → **"Settings"** 탭
2. **"Deploy"** 섹션 → **"Start Command"** 항목에 아래 명령어 입력:

```
uvicorn api.main:app --host 0.0.0.0 --port $PORT
```

3. **Save** 클릭 → 자동 재배포 시작

> `$PORT`는 Railway가 자동으로 지정하는 포트 번호입니다. 직접 숫자로 바꾸지 마세요.

### B-6) Notion 자동 동기화 Cron 설정 (약 10분)

현재 Vercel에서는 매일 새벽 4시(KST)에 노션을 자동 동기화합니다.
Railway에서 동일하게 설정합니다.

1. Railway 대시보드 → 프로젝트 화면에서 **"+ New"** 클릭
2. **"Cron Job"** 선택
3. 아래와 같이 설정:
   - **Name**: `노션 자동 동기화`
   - **Schedule**: `0 19 * * *`
     > 매일 UTC 19:00 = 한국시간(KST) 새벽 4:00
   - **Command**:
     ```
     curl -X GET https://[Railway도메인]/api/cron/sync_notion
     ```
4. 저장

> Railway 도메인은 프로젝트 → **"Settings"** → **"Domains"** 에서 확인합니다.
> 확인 후 위 명령어의 `[Railway도메인]` 부분을 실제 도메인으로 교체하세요.

### B-7) 화면 정상 동작 확인 ⚠️ (약 10분)

Vercel은 위젯·대시보드 화면 파일을 플랫폼이 직접 제공했지만,
Railway에서는 챗봇 앱이 직접 제공합니다. 아래 항목을 반드시 확인하세요.

| 접속 주소 | 기대 결과 |
|----------|----------|
| `https://[Railway도메인]/widget.html` | 챗봇 위젯 화면이 열려야 함 |
| `https://[Railway도메인]/admin/login.html` | 로그인 화면이 열려야 함 |
| `https://[Railway도메인]/admin` | 로그인 화면으로 이동해야 함 |

> **`/admin` 경로 참고**: Railway에서 `/admin` 직접 접속 시 대시보드 화면이 먼저 뜰 수 있습니다.
> 로그인이 안 된 상태라면 자동으로 로그인 화면으로 이동하므로 **기능상 문제는 없습니다.**

### B-8) 관리자 계정 확인 (약 5분)

Railway와 Vercel은 **동일한 Supabase DB**를 사용하므로, 기존 관리자 계정이 그대로 유지됩니다.

1. `https://[Railway도메인]/admin/login.html` 접속
2. 기존 이메일·비밀번호로 로그인 시도
3. 로그인 성공하면 완료 — 계정을 새로 만들 필요 없습니다

> 로그인이 안 된다면 Railway 대시보드 → 프로젝트 → **"Shell"** 탭에서
> `python create_admin_account.py` 명령으로 계정을 새로 생성하세요.

---

### B-9) 노션 임베드 URL 교체 (약 5분)

노션 페이지에 임베드된 챗봇 위젯 주소를 Railway 도메인으로 바꿉니다.

1. 노션 페이지 열기 → 챗봇 임베드 블록 찾기
2. 블록 우클릭 (또는 블록 오른쪽 **"..."**) → **"원본 편집"** 또는 **"Edit"**
3. URL 변경:
   - **변경 전**: `https://[기존Vercel도메인]/widget.html`
   - **변경 후**: `https://[Railway도메인]/widget.html`
4. 저장 후 챗봇 위젯이 정상적으로 열리는지 확인

### B-10) 기존 Vercel 프로젝트 정리

Railway에서 모든 기능이 정상 동작하는 것을 **최소 2~3일 확인한 뒤** 진행합니다.

삭제 전 최종 체크리스트:
- [ ] Railway에서 챗봇 질문 응답 정상 확인
- [ ] Railway에서 대시보드 로그인 및 전체 메뉴 정상 확인
- [ ] Railway Cron이 최소 1회 이상 정상 실행됐는지 확인 (대시보드 크론 배너로 확인)
- [ ] 노션 임베드 URL이 Railway 주소로 교체됐는지 확인

위 항목 모두 확인 후:
[vercel.com](https://vercel.com) → 기존 프로젝트 → **Settings** → 맨 아래 **"Delete Project"**
